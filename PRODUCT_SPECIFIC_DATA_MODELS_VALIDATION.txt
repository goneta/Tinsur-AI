Product-specific data models validation
=======================================

Scope implemented
-----------------
This implementation adds tenant-scoped, product-versioned detail attachment records so the platform no longer needs to rely only on generic Quote, Policy, or Claim JSON payloads for product-specific insurance data. The core Quote, Policy, and Claim models remain generic, while product-specific detail records attach to them through one-to-one relationships.

Implemented tables and model responsibilities:

1. product_application_details
   - Attaches to quotes through quote_id.
   - Links every record to company_id and insurance_product_versions.id through product_version_id.
   - Optionally links to product_quote_wizard_schemas.id through quote_wizard_schema_id.
   - Stores schema_version, validation_schema_ref, application_data, validation_status, validation_errors, created_at, and updated_at.

2. product_rating_snapshots
   - Attaches to policies through policy_id.
   - Links every record to company_id and insurance_product_versions.id through product_version_id.
   - Stores schema_version, validation_schema_ref, rating_data, validation_status, validation_errors, created_at, and updated_at.

3. product_claim_details
   - Attaches to claims through claim_id.
   - Links every record to company_id and insurance_product_versions.id through product_version_id.
   - Stores schema_version, validation_schema_ref, claim_data, validation_status, validation_errors, created_at, and updated_at.

Focused test coverage
---------------------
The new focused tests validate that each new table contains tenant, product-version, schema version, validation status, validation errors, and product-specific payload fields. The tests also validate Pydantic DTO behavior for non-empty schema versions, required product-specific payloads, and structured validation errors for invalid records. Relationship tests confirm that Quote, Policy, and Claim retain their generic details payloads while attaching separate product-specific application, rating, and claim-detail objects.

Remaining risks and follow-up recommendations
---------------------------------------------
1. The new tables and schemas provide the persistence and validation contract, but existing quote acquisition, rating, policy binding, and claim intake services should be updated in a later milestone to persist these records automatically during normal workflows.
2. The schema currently validates structural requirements at the DTO level. Deep JSON Schema validation against product_quote_wizard_schemas.validation_schema or future claim-specific schemas should be added at the service layer so validation errors can be normalized before records are stored.
3. No public API endpoints were added in this focused model milestone. Existing endpoints can remain backward-compatible, and new endpoint or service integration should be introduced deliberately once the workflow touchpoints are selected.
