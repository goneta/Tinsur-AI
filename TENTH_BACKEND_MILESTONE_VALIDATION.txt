## Implementation notes

The new hardening service treats user prompts as untrusted input and performs lightweight deterministic screening before external AI provider calls. It detects common prompt-injection patterns, such as attempts to override system instructions or exfiltrate hidden prompts, and returns a safe refusal message for blocked prompts. It also redacts common sensitive values, including emails and key-like secrets, before prompts or errors are stored in observability metadata.

The AI observability payload records tenant scope, user identifier, agent/action/route labels, prompt hash, redacted prompt preview, safety status, history count, provider/model information, retry attempts, fallback usage, and redacted errors. This gives operators useful production diagnostics without storing raw prompts by default. The retry helper is intentionally bounded and defaults to short exponential backoff with jitter to avoid unbounded provider loops.

## Remaining risks and follow-up recommendations

This milestone deliberately avoids a broader AI governance buildout. The deterministic prompt-safety patterns should be treated as a first production-hardening layer rather than a complete adversarial AI defense. Future milestones should consider provider-specific content safety APIs, tenant-configurable safety policy levels, structured telemetry export to the production logging stack, rate limiting by tenant/user/route, and red-team regression suites for prompt-injection variants. Streaming WebSocket fallback now uses sanitized prompts, but true token-level streaming observability remains limited to final usage logging because the current router interface does not expose per-token metadata.
